ALBOT — FUTURE SCOPE & PLANNED FEATURES
========================================
(Feature #10 from the original list — Real-Time Web Search Integration into Chat —
has already been implemented inside the existing web search functionality and is
therefore excluded from this document.)


─────────────────────────────────────────────────────────────────────────────────
1. ADAPTIVE RESEARCH LOOP (Deep Research)
─────────────────────────────────────────────────────────────────────────────────

The Problem: The current pipeline is linear — plan is generated once, then executed
mechanically regardless of what findings reveal. Real researchers adapt.

What to Build: After each step completes, the orchestrator should pass the step's
findings back to the planner and ask: "Given what we just learned, which remaining
steps are still needed, which are now redundant, and what NEW steps should be
added?" The planner returns a PlanRevision object containing steps to skip, steps
to promote (execute earlier), and new steps to inject. The step queue becomes a
dynamic priority queue rather than a static list.

Integration points: ResearchPlanner gets a new revise_plan() async method. The
orchestrator's research loop becomes a "while queue" pattern instead of
"for step in plan.steps". The context graph gets a PLAN_REVISED edge type linking
the original plan node to the new step nodes, preserving full provenance of why
the plan changed. The frontend DeepResearchPanel.tsx would show a "Plan updated"
badge when steps are dynamically added.


─────────────────────────────────────────────────────────────────────────────────
2. MULTI-AGENT RESEARCH CREW
─────────────────────────────────────────────────────────────────────────────────

The Problem: A single LLM agent doing all roles — planner, scraper, extractor,
critic, synthesizer — creates quality bottlenecks and a single point of failure.

What to Build: A crew of specialised async agents running concurrently:

- PlannerAgent: Generates and revises the research plan (existing ResearchPlanner,
  enhanced).
- SearchAgent (xN): Executes web searches in parallel, one per research step.
  Currently only 1 search runs at a time per step.
- ExtractionAgent (xN): Runs RLM extraction. Currently 1 per page serially.
- CriticAgent: After each step synthesis, scores the synthesis on 5 dimensions
  (completeness, accuracy, depth, relevance, source diversity) and flags it for
  re-research if below threshold. Uses LLMJudge which already exists in
  llm_judge/llm_judge.py.
- FactCheckerAgent: Takes every claim with a citation from the synthesis and
  verifies the citation URL actually supports the claim by re-scraping and
  re-querying the source.
- SynthesisAgent: Existing synthesize_final enhanced with iterative refinement —
  generates the report, then critiques it against the original query, then
  regenerates weak sections.

Integration: Create backend/core/deep_research/research_crew.py. The
DeepResearchOrchestrator delegates to ResearchCrew.run() instead of orchestrating
directly. Each agent is a lightweight class with a single async def run() method
and an internal asyncio.Queue for input/output.


─────────────────────────────────────────────────────────────────────────────────
3. GRAPHRAG — COMMUNITY SUMMARISATION AND MULTI-HOP TRAVERSAL
─────────────────────────────────────────────────────────────────────────────────

The Problem: Your knowledge graph in ArangoDB is built and visualised, but
retrieval only uses it for edge-hop proximity scoring (β weight in
RetrievalWeights). The graph's structural intelligence — clusters of related
concepts — is completely untapped for query answering.

What to Build (inspired by Microsoft's GraphRAG paper):

Part A — Community Detection and Summaries: After any ingestion, run Louvain
community detection (python-louvain is already in requirements.txt) on the
ArangoDB knowledge graph. For each detected community, use the LLM to generate a
CommunitySummary — a 200-word paragraph describing what concepts this cluster
contains and how they relate. Store these in a new community_summaries collection
in ArangoDB. During query time, embed the query and find the top-K most relevant
community summaries first. These provide high-level context before retrieving
specific chunks.

Part B — Multi-Hop Query Expansion: When the retrieval engine gets a query, it
currently retrieves nodes within 2 hops of matched nodes. Extend this to run a
multi_hop_subgraph_query in ArangoDB AQL that starts from matched nodes and
follows edges of types REFERENCES, ENTITY_SHARED, DERIVED_FROM up to 3 hops,
collecting all intermediate node content. Feed this traversal context to the LLM
alongside the direct retrieval results.

Integration: backend/core/graph/graph_builder.py gets a build_community_summaries()
method triggered after every ingestion. backend/core/retrieval/retrieval_engine.py
gets a graph_rag_retrieve() path that the orchestrator selects when query
complexity is high (detected by question word count > 15 or presence of
causal/comparative language).


─────────────────────────────────────────────────────────────────────────────────
4. HYDE — HYPOTHETICAL DOCUMENT EMBEDDINGS
─────────────────────────────────────────────────────────────────────────────────

The Problem: When a user asks "What are the tradeoffs between RAG and fine-tuning?",
the embedding of that question looks nothing like the embedding of an academic paper
that answers it. The semantic gap between question-style text and answer-style text
degrades retrieval precision.

What to Build: Before doing any vector retrieval, the LLM generates a hypothetical
ideal answer to the query — roughly 150 words, written in the style of a document
that would perfectly answer it. That hypothetical answer is then embedded (not the
original query). The resulting embedding sits in "answer space" rather than
"question space" and retrieves far more relevant documents. Run both embeddings
(original query + hypothetical answer), retrieve top-K for each, then union and
re-rank by score.

Integration: Add a hyde_generate() method to EmbeddingEngine in embedding_engine.py.
In retrieval_engine.py, add a use_hyde: bool parameter to the retrieve method,
defaulting to True for queries longer than 8 words. The LLM call for HyDE uses
async_complete() with max_tokens=250 and temperature=0.7 — it is intentionally
creative to cast a wider retrieval net.


─────────────────────────────────────────────────────────────────────────────────
5. STREAMING RESEARCH REPORT WITH PROGRESSIVE RENDERING
─────────────────────────────────────────────────────────────────────────────────

The Problem: The final synthesis call (synthesize_final) waits 30-60 seconds and
then dumps the entire 5000-word report at once. The user sees a spinner, then a
wall of text.

What to Build: Use the streaming API from each LLM provider (OpenAI's stream=True,
Anthropic's stream=True) to yield tokens progressively. Add async_stream() to
LLMRouter that returns an AsyncGenerator[str, None]. In synthesize_final, replace
the single async_complete() call with async_stream() and yield each token through
a new on_token callback. The orchestrator passes this callback through to the SSE
stream endpoint (/research/{id}/stream). The frontend DeepResearchPanel.tsx appends
tokens to a live-rendered markdown div. Users see the report being written in real
time.

Integration: LLMRouter.async_stream() yields str tokens. RecursiveResearchAgent
gets synthesize_final_streaming() taking an on_token: Callable[[str], Awaitable[None]]
parameter. The _run() background task in main.py passes a callback that pushes
tokens into the SSE event queue. The existing event_generator() SSE function picks
up "token" event types.


─────────────────────────────────────────────────────────────────────────────────
6. CROSS-SESSION RESEARCH MEMORY AND KNOWLEDGE ACCUMULATION
─────────────────────────────────────────────────────────────────────────────────

The Problem: Every deep research session starts from zero. If you researched
"transformer attention mechanisms" yesterday, today's session on "efficient
transformers" has no access to those findings.

What to Build: After a research session completes, run a post-processing pipeline
that:
1. Extracts all Finding objects with importance >= 0.7
2. Embeds each finding using EmbeddingEngine
3. Stores them as MemoryFragment objects in ArangoDB with
   namespace="research_knowledge", tagged with the original query, session ID,
   and source domains
4. Builds edges in the knowledge graph linking these research findings to any
   existing KnowledgeAtom nodes that cover the same entities

At the start of a new research session, the orchestrator queries the
research_knowledge namespace for fragments semantically similar to the new query.
These fragments are injected into the synthesize_final prompt as "Prior Research
Context", labelled with their source sessions. The planner also receives these
prior findings and can skip steps that are already well-covered.

Integration: New ResearchMemoryIndexer class in backend/core/deep_research/.
Called from the _run() background task in main.py after orchestrator.run_research()
completes. The memory manager's existing store_fragment() and search_fragments()
methods handle persistence and retrieval.


─────────────────────────────────────────────────────────────────────────────────
7. AUTOMATED KNOWLEDGE GRAPH POPULATION FROM RESEARCH FINDINGS
─────────────────────────────────────────────────────────────────────────────────

The Problem: Deep research generates hundreds of structured findings with entities,
relationships, and citations — but none of this gets added to the ArangoDB
knowledge graph. The graph only grows when files are explicitly ingested.

What to Build: After each research session, run an entity-relationship extraction
pass over all findings using a structured LLM prompt. The prompt asks the LLM to
identify: (a) named entities (models, papers, authors, organisations, concepts),
(b) relationships between them (X outperforms Y, X is-a-variant-of Y, X
proposed-by Y), (c) confidence scores. These become KnowledgeAtom nodes and
GraphEdge objects in ArangoDB, linked to their source research session. Over time,
the knowledge graph becomes a rich, automatically-curated database of research
findings.

Integration: New ResearchGraphIngester in backend/core/deep_research/. Uses the
existing graph_builder.py add_node() and add_edge() APIs. The entity extraction
prompt is a structured JSON schema-constrained call using async_complete(). Runs
as a non-blocking background task after session completion so it does not delay the
user's result.


─────────────────────────────────────────────────────────────────────────────────
8. CONTRADICTION DETECTION AND CLAIM RECONCILIATION
─────────────────────────────────────────────────────────────────────────────────

The Problem: When 15 sources are scraped and hundreds of findings extracted,
conflicting information is silently averaged away in synthesis. A paper from 2021
might say X achieves 85% accuracy; a 2024 paper might say X only achieves 72% on
a fairer benchmark. The synthesiser blends these without flagging the contradiction.

What to Build: Before synthesize_step(), run a ContradictionDetector that:
1. Groups findings by semantic similarity (using embedding cosine similarity,
   threshold 0.75)
2. Within each cluster, asks the LLM: "Do any of these findings contradict each
   other? If yes, explain the nature of the contradiction and which finding is more
   likely correct (consider publication year, source authority, methodology quality)."
3. Returns ContradictionReport objects with: the contradicting findings, the nature
   of conflict, and the reconciliation recommendation
4. Injects ContradictionReport objects into the synthesize_step prompt with a
   special section: "Known Contradictions in the Evidence" — forcing the
   synthesiser to explicitly address them
5. Stores contradiction edges in the context graph using the existing CONTRADICTS
   edge type (already defined in ResearchEdgeType but currently never populated)

Integration: New ContradictionDetector class in backend/core/deep_research/.
Called from the orchestrator between extraction and step synthesis. The frontend
can display contradiction counts as a quality signal next to each step.


─────────────────────────────────────────────────────────────────────────────────
9. INTELLIGENT SOURCE TRUST SCORING
─────────────────────────────────────────────────────────────────────────────────

The Problem: A preprint from an unknown lab and a Nature paper both get
relevance_score: 0.0 in SourceInfo. The web scraper has domain priority but no
per-source quality assessment.

What to Build: A SourceTrustScorer that computes a multi-factor trust score for
each scraped source:
- Domain authority (0-40 pts): Priority domain list -> 40 pts, unknown domain ->
  5 pts
- Publication venue (0-25 pts): Detect if content mentions NeurIPS/ICML/ICLR/
  Nature -> 25 pts; arXiv -> 15 pts; blog -> 5 pts
- Citation count proxy (0-20 pts): Scrape api.semanticscholar.org/graph/v1/paper/
  search?query=... with the paper title to get real citation counts (async, with
  timeout)
- Temporal relevance (0-15 pts): Extract publication year from content; papers
  <= 2 years old -> 15 pts; <= 5 years -> 10 pts; older -> 5 pts

The trust score feeds into finding importance weighting: findings from high-trust
sources get their importance boosted by score/100. The final synthesis prompt
receives sources sorted by trust score, and the LLM is instructed to weight
high-trust findings more heavily in its claims.

Integration: SourceTrustScorer sits in backend/core/deep_research/. Called after
scraper.scrape_urls(), before RLM extraction. The SourceInfo model gets a
trust_score: float field. The frontend shows trust badges on sources.


─────────────────────────────────────────────────────────────────────────────────
10. CODE REPOSITORY INGESTION AND CODE-AWARE RAG
─────────────────────────────────────────────────────────────────────────────────

The Problem: The multimodal processor handles PDFs, images, audio, video — but
code repositories are not treated as first-class knowledge sources. Ingesting a
GitHub repo just treats code files as text, losing all structural semantics
(function signatures, call graphs, imports).

What to Build: A CodeRepositoryIngester in backend/core/ingestion/ that:
1. Walks a git repository (or a zip upload) and parses each file using tree-sitter
   (language-agnostic AST parser)
2. Extracts: function definitions with docstrings, class hierarchies, import
   graphs, test cases
3. Creates KnowledgeAtom nodes at three resolutions: file level (COARSE), function
   level (MID), docstring/signature level (FINE)
4. Builds GraphEdge relationships: CALLS, IMPORTS, IMPLEMENTS, TESTS
5. Generates embedding-friendly textual descriptions of each function:
   "Function embed_text in embedding_engine.py takes a list of strings and returns
   a numpy array of shape (N, 1024). Uses sentence-transformers E5-Large model.
   Called by retrieval_engine.py."

For RAG queries about code ("How does the embedding engine work?", "What calls the
web search?"), the graph traversal gives structurally correct, call-graph-informed
answers instead of naive text similarity.


─────────────────────────────────────────────────────────────────────────────────
11. LLM-POWERED INGESTION QUALITY CONTROL
─────────────────────────────────────────────────────────────────────────────────

The Problem: When a PDF is ingested, there is no quality check. A corrupted PDF,
a scanned-image PDF (no text), or a 1000-page book are all ingested with the same
pipeline, often producing garbage embeddings.

What to Build: A pre-ingestion ContentQualityGate that runs before any embedding:
1. Text density check: If extracted text / total pages < 50 words per page, flag
   as "likely scanned image PDF" and trigger OCR via Tesseract (already in
   requirements).
2. Language detection: If content is not English and no translation is configured,
   skip or flag.
3. Duplicate detection: Compute a MinHash signature of the document. Check against
   existing signatures in SQLite. If Jaccard similarity > 0.9, reject as
   near-duplicate and notify user.
4. Content classification: Quick LLM call (50 tokens) to classify the document as:
   academic paper, technical documentation, news article, blog post, book, etc.
   This classification feeds into optimal chunking strategy (academic papers ->
   larger chunks; news articles -> smaller chunks).
5. Automatic metadata extraction: Extract title, authors, year, abstract, key terms
   in a single LLM call. Store as node properties in ArangoDB.


─────────────────────────────────────────────────────────────────────────────────
12. PROACTIVE RESEARCH SUGGESTIONS
─────────────────────────────────────────────────────────────────────────────────

The Problem: After a deep research session completes, the user has a report but no
guidance on what to investigate next. The system knows a lot — it has all the
findings, the source graph, the identified contradictions — but stays silent.

What to Build: After synthesize_final() completes, run a ResearchGapAnalyzer:
1. Asks the LLM: "Given this research on [query] and these findings, what are the
   3 most important follow-up research questions that would deepen understanding?
   What specific knowledge gaps were identified?"
2. Returns structured FollowUpQuestion objects:
   {question: str, rationale: str, estimated_complexity: "quick|medium|deep",
    suggested_first_query: str}
3. These are stored in the session result and displayed in the frontend as a
   "Research Further" section with one-click buttons that start a new research
   session pre-populated with the follow-up question.

Integration: ResearchGapAnalyzer is a single async method in rlm_engine.py. Called
at the end of _finalize(). The ResearchResult model gets a
follow_up_questions: List[FollowUpQuestion] field. The frontend
DeepResearchPanel.tsx renders them as suggestion cards after the report.


─────────────────────────────────────────────────────────────────────────────────
13. RETRIEVAL-AUGMENTED FINE-TUNING DATA GENERATION
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot generates high-quality Q&A pairs implicitly — every query +
retrieved context + synthesised answer is a perfect training example — but this
data is thrown away.

What to Build: A TrainingDataExporter that:
1. After each successful chat turn (where the LLM judge scored quality >= 7/10),
   packages: {instruction: query, context: retrieved_chunks, output: answer} into
   JSONL format.
2. After each deep research session, packages: {instruction: research_query,
   context: all_findings_text, output: final_report} into fine-tuning format.
3. Provides a GET /export/training-data endpoint that returns the accumulated JSONL.
4. Optionally pushes to Hugging Face Hub via the existing HF_TOKEN in .env.

This turns Albot into a self-improving dataset generator for domain-specific
fine-tuning.


─────────────────────────────────────────────────────────────────────────────────
14. SEMANTIC SEARCH OVER RESEARCH HISTORY
─────────────────────────────────────────────────────────────────────────────────

The Problem: A user runs 50 research sessions over months. Finding a specific past
report requires remembering the exact query. There is no way to search across past
research by concept.

What to Build: Index every completed research session's report and findings into a
dedicated ArangoDB collection with full embeddings. Expose
GET /research/search?q=... that does semantic search over past reports — not by
query string but by conceptual similarity. Results show: session date, original
query, report preview, and similarity score. The frontend sidebar gets a
"Past Research" section with semantic search. This reuses the existing
EmbeddingEngine.embed_query() and ArangoDB vector search infrastructure that is
already built for the RAG knowledge base.

─────────────────────────────────────────────────────────────────────────────────
15. CONVERSATIONAL DATA ANALYST
─────────────────────────────────────────────────────────────────────────────────

The Problem: Users can ingest CSVs and Excel files, but they become dumb text
chunks. There is no way to ask "What's the average revenue by quarter?" and get
an actual computed answer with a chart. The current pipeline treats structured
data the same as unstructured prose.

What to Build: A DataAnalystSkill that detects when a user's query references
an ingested tabular dataset (or when they upload one mid-conversation). Instead
of doing RAG retrieval, the skill:
1. Loads the dataset into a pandas DataFrame in a sandboxed environment.
2. Uses the LLM to translate the natural-language question into a pandas
   operation chain (e.g., df.groupby('region')['revenue'].mean()).
3. Executes the generated code in a restricted sandbox (RestrictedPython or
   a subprocess jail) — never eval() on raw LLM output.
4. If the result is numeric/tabular, renders it as a formatted markdown table.
5. If the query implies visualization ("show me a chart", "plot", "trend"),
   generates a matplotlib/plotly figure, saves it as a PNG, and returns it
   inline in the chat as a base64 image.
6. Supports follow-up queries: "Now filter that to only Q4" retains the
   DataFrame state across turns within the same chat session.
7. Automatically profiles new datasets on upload: row count, column types,
   missing values, basic statistics, and stores a DataProfile object that
   the LLM uses to write better queries.

Integration: New backend/core/skills/data_analyst.py. The main orchestrator
detects tabular intent via keyword heuristics ("average", "sum", "plot",
"correlate", column names found in ingested datasets) and routes to this skill
instead of the standard RAG path. Frontend gets a new DataResultCard component
that renders tables and inline charts.


─────────────────────────────────────────────────────────────────────────────────
16. DOCUMENT GENERATOR — REPORTS, PROPOSALS, AND BRIEFS
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot is excellent at answering questions but cannot produce
structured deliverables. A user who researches "competitive landscape for
edge AI chips" cannot say "now write me a 10-page executive brief from that."

What to Build: A DocumentGeneratorSkill triggered by explicit commands
("/generate report", "/write proposal", "/draft brief") or detected intent.
The skill:
1. Takes a document type (report, proposal, executive brief, literature review,
   technical spec, meeting summary) and an optional topic or prior chat context.
2. Uses a type-specific structural template: e.g., executive briefs get
   Executive Summary → Key Findings → Competitive Analysis → Recommendations →
   Appendix.
3. For each section, runs a focused LLM call with relevant context pulled from
   the knowledge base, recent chat history, or prior deep research sessions.
4. Assembles the sections into a coherent document with consistent tone,
   cross-references between sections, and a table of contents.
5. Exports to multiple formats: Markdown (inline in chat), PDF (via
   weasyprint or reportlab), DOCX (via python-docx), and HTML.
6. Provides a download link in the chat interface.

Integration: New backend/core/skills/document_generator.py. New API endpoint
POST /generate/document with body {type, topic, context_chat_id, format}.
Frontend gets a DocumentExportModal component with format selection and
preview.


─────────────────────────────────────────────────────────────────────────────────
17. ADAPTIVE LEARNING ENGINE — QUIZZES, FLASHCARDS, AND SPACED REPETITION
─────────────────────────────────────────────────────────────────────────────────

The Problem: Users ingest textbooks, papers, and lecture notes for studying,
but Albot only supports free-form Q&A. There is no active learning support —
no quizzes, no flashcards, no way to test comprehension.

What to Build: A LearningEngineSkill that transforms ingested content into
active learning tools:
1. Quiz Generation: Given a document or topic, the LLM generates N questions
   at a specified difficulty (easy/medium/hard) and type (multiple choice,
   true/false, short answer, fill-in-the-blank). Each question includes the
   correct answer and an explanation referencing the source material.
2. Flashcard Decks: Extracts key concepts → generates front/back flashcard
   pairs. Stored in a flashcards SQLite table. Supports export to Anki format
   (via genanki library).
3. Spaced Repetition Scheduler: Implements SM-2 algorithm. When a user reviews
   a flashcard and rates difficulty (1-5), the scheduler computes the next
   review date. Cards due for review are surfaced proactively at the start of
   chat sessions.
4. Knowledge Gap Detection: After a quiz, analyzes which topics the user got
   wrong, maps them back to source documents, and suggests targeted re-reading
   sections or generates focused follow-up quizzes on weak areas.
5. Progress Dashboard: Track accuracy over time, mastery per topic, streak
   counts.

Integration: New backend/core/skills/learning_engine.py. New SQLite tables:
flashcard_decks, flashcards, quiz_attempts, review_schedule. Frontend gets
QuizPanel and FlashcardViewer components. API endpoints:
POST /learn/generate-quiz, POST /learn/generate-flashcards,
GET /learn/due-reviews, POST /learn/submit-review.


─────────────────────────────────────────────────────────────────────────────────
18. VOICE INTERFACE — SPEECH INPUT AND AUDIO OUTPUT
─────────────────────────────────────────────────────────────────────────────────

The Problem: Interaction is text-only. Users cannot speak to Albot or listen
to responses, which limits usability for hands-free workflows, accessibility,
and mobile use.

What to Build:
Input: Add a microphone button to the chat input. On click, the browser's
MediaRecorder API captures audio, which is sent as a WAV blob to a new
POST /voice/transcribe endpoint. The backend uses the already-loaded Whisper
model (MultimodalProcessor already has it) to transcribe. The transcript is
inserted into the chat input field for user review before sending, or
auto-sent if the user enables "auto-send" mode.

Output: After a response is generated, an optional "Read aloud" button appears
next to the response. On click, the response text is sent to
POST /voice/synthesize. The backend uses a TTS engine — either edge-tts
(free, Microsoft Azure voices, pip installable), Coqui TTS (local,
open-source), or OpenAI TTS API. Returns an audio stream that the frontend
plays via an <audio> element.

Streaming mode: For long responses, use chunked TTS — split the response into
sentences, synthesize each as a separate audio segment, and stream them as
they complete so playback starts within 1-2 seconds.

Integration: New backend/core/skills/voice_interface.py. Two new API endpoints.
Frontend gets VoiceInputButton and AudioPlaybackButton components. Whisper
model is shared with the existing MultimodalProcessor instance to avoid double
memory usage.


─────────────────────────────────────────────────────────────────────────────────
19. MULTI-LANGUAGE PIPELINE — AUTO-DETECT, TRANSLATE, RESPOND
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot only works well in English. A user who ingests a French PDF
or asks a question in Spanish gets degraded retrieval (English embeddings
perform poorly on non-English text) and inconsistent response language.

What to Build: A LanguagePipeline middleware that wraps the entire query and
response flow:
1. Language Detection: Use langdetect or fasttext-langdetect on every incoming
   query. If the detected language is not English, flag the query.
2. Query Translation: Translate non-English queries to English before
   embedding and retrieval (using an LLM translation call or Helsinki-NLP
   MarianMT models which are small and local).
3. Retrieval proceeds in English (where embeddings are strongest).
4. Response Translation: After the LLM generates the English response,
   translate it back to the user's detected language.
5. Bilingual Mode: Optionally return both the English response and the
   translated response, useful for language learners.
6. Ingestion Translation: When a non-English document is ingested, run
   translation at chunk level before embedding. Store both original and
   translated text. The knowledge atom gets a language: str field and a
   translated_content: Optional[str] field.
7. Per-user language preference stored in session settings, overriding
   auto-detection.

Integration: LanguagePipeline as a middleware class in
backend/core/language/translation_pipeline.py. Wraps LLMRouter.complete()
calls. The orchestrator checks query language before routing. Frontend gets a
language selector dropdown in settings.


─────────────────────────────────────────────────────────────────────────────────
20. GITHUB/GITLAB INTEGRATION — PR REVIEW, ISSUE ANALYSIS, CODEBASE Q&A
─────────────────────────────────────────────────────────────────────────────────

The Problem: Developers use Albot for research but switch to a different tool
for code review. There is no bridge between the knowledge base and a team's
actual codebase or issue tracker.

What to Build: A GitIntegrationSkill:
1. Repository Connection: User provides a GitHub/GitLab repo URL + personal
   access token. The backend clones (shallow, depth=1) or uses the API to
   fetch file trees and contents.
2. PR Review Mode: Given a PR number, the skill fetches the diff via GitHub
   API (GET /repos/{owner}/{repo}/pulls/{num}/files), parses each changed
   file, and asks the LLM to review for: bugs, security vulnerabilities,
   style inconsistencies, missing tests, and performance concerns. Returns
   structured ReviewComment objects with file, line number, severity, and
   suggestion.
3. Issue Analysis: Fetches open issues, clusters them by topic (using
   embeddings), and generates a prioritized summary: "You have 12 open bugs,
   5 feature requests, and 3 documentation issues. The most urgent cluster
   is around authentication failures (issues #42, #47, #51)."
4. Codebase Q&A: After cloning, runs the CodeRepositoryIngester (Feature #10
   from the existing scope) and enables RAG queries against the code:
   "How does error handling work in the API layer?"
5. Commit Changelog Generator: Given a date range or tag range, fetches
   commits and generates a human-readable changelog grouped by category
   (features, fixes, refactors).

Integration: New backend/core/integrations/git_integration.py. Uses PyGithub
or httpx for API calls. Frontend gets a GitHubConnectPanel in settings and a
PRReviewPanel in the chat interface.


─────────────────────────────────────────────────────────────────────────────────
21. SCHEDULED RESEARCH DIGESTS — AUTOMATED TOPIC MONITORING
─────────────────────────────────────────────────────────────────────────────────

The Problem: Research is always user-initiated. A user tracking "edge AI
hardware" must manually run a new deep research session every week to stay
updated. There is no proactive monitoring.

What to Build: A ResearchScheduler that lets users create "watch topics":
1. Users define a topic and a schedule (daily, weekly, biweekly).
2. At the scheduled time, a background task runs a lightweight version of
   deep research: 2-3 search steps, focused on recency (search queries
   include "2026", "latest", "recent").
3. The system compares new findings against the user's existing research
   memory (Feature #6). Only genuinely new information is kept.
4. A digest is generated: a 500-word summary of what's new, with source
   links and a diff-style highlight of what changed since the last digest.
5. Delivery options: in-app notification (new chat message), email
   (via SendGrid/SMTP), or webhook (POST to a user-defined URL — useful
   for Slack/Discord/Teams integration).
6. The digest is stored as a regular chat message so users can ask follow-up
   questions against it.

Integration: New backend/core/scheduling/research_scheduler.py using APScheduler.
New SQLite tables: watch_topics, digest_history. API endpoints:
POST /schedule/watch, GET /schedule/topics, DELETE /schedule/watch/{id}.
Frontend gets a WatchTopicPanel in the sidebar.


─────────────────────────────────────────────────────────────────────────────────
22. DOMAIN SPECIALIST MODES — LEGAL, MEDICAL, FINANCIAL, ACADEMIC
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot uses generic prompts for all queries. A lawyer analyzing a
contract, a doctor reviewing clinical notes, and a finance analyst reading
10-K filings all get the same generic RAG treatment. Domain-specific
terminology, reasoning patterns, and output formats are lost.

What to Build: A DomainModeSelector that lets users switch Albot into a
specialist mode. Each mode configures:
1. System Prompt Override: Domain-tailored system prompts that set tone,
   terminology expectations, and reasoning style. E.g., Legal mode: "You are
   a legal research assistant. Cite statutes by jurisdiction. Flag ambiguous
   clauses. Never provide legal advice — frame everything as analysis."
2. Custom Retrieval Weights: Legal mode boosts exact keyword matches (statute
   numbers are precise). Medical mode boosts recency (treatment guidelines
   change). Financial mode boosts numerical precision.
3. Domain-Specific Post-Processing:
   - Legal: Extract clause references, flag risk levels (low/medium/high),
     generate clause-by-clause comparison tables.
   - Medical: Extract drug names, dosages, contraindications. Flag
     off-label usage. Add safety disclaimers automatically.
   - Financial: Extract financial metrics (P/E, EBITDA, margins), generate
     comparison tables, compute ratios from raw data.
   - Academic: Auto-generate citation lists, identify methodology types,
     flag statistical claims that lack confidence intervals.
4. Domain Glossary: Each mode has a curated terminology list that helps the
   embedding engine disambiguate domain-specific terms.
5. Output Templates: Each mode has preferred output formats (legal memos,
   clinical summaries, investment briefs, literature reviews).

Integration: New backend/core/skills/domain_modes.py with a DomainMode enum
and DomainConfig dataclass. The orchestrator loads the active mode's config
before each query. Frontend gets a mode selector dropdown in the header bar
with icons for each domain.


─────────────────────────────────────────────────────────────────────────────────
23. MULTI-USER COLLABORATIVE WORKSPACES
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot is single-user. A team of researchers cannot share a
knowledge base, collaborate on research sessions, or see each other's findings.
Every user operates in isolation.

What to Build: A WorkspaceSystem with:
1. User Authentication: Basic user accounts (email + password, or OAuth via
   Google/GitHub). JWT-based session tokens.
2. Workspaces: Named collections of knowledge bases, chat sessions, and
   research history. A workspace can be private (single user) or shared
   (team).
3. Shared Knowledge Bases: Multiple users ingest documents into the same
   ArangoDB collections, scoped by workspace_id. All team members query the
   same knowledge.
4. Collaborative Research: A deep research session can be shared — all team
   members see real-time progress and can view the final report. Comments
   and annotations on research findings.
5. Permission Levels: Owner (full control), Editor (can ingest, query, and
   run research), Viewer (read-only access to existing research and chats).
6. Activity Feed: A workspace-level activity log showing who ingested what,
   who ran which research, and recent queries — providing team awareness
   without real-time presence tracking.
7. Conflict-Free Ingestion: Multiple users ingesting simultaneously use
   optimistic concurrency in ArangoDB (document revision checks) to prevent
   data corruption.

Integration: New backend/core/auth/ module with user_manager.py and
workspace_manager.py. New SQLite tables: users, workspaces, workspace_members,
activity_log. All existing API endpoints get an Authorization header check.
Frontend gets LoginPage, WorkspaceSwitcher, and TeamActivityPanel components.


─────────────────────────────────────────────────────────────────────────────────
24. CITATION MANAGER AND MULTI-FORMAT EXPORT
─────────────────────────────────────────────────────────────────────────────────

The Problem: Deep research produces source lists, but they are plain URLs.
Researchers need properly formatted citations (APA, MLA, Chicago, BibTeX,
IEEE) and there is no way to export them into reference managers like Zotero
or Mendeley.

What to Build: A CitationManagerSkill:
1. Metadata Extraction: For every source URL in a research session, attempt
   to extract full bibliographic metadata: title, authors, journal/venue,
   year, DOI, volume, issue, pages. Use multiple strategies:
   - Scrape <meta> tags (citation_title, citation_author, citation_doi)
   - Query CrossRef API (api.crossref.org/works?query=...) for DOI lookup
   - Query Semantic Scholar API for academic papers
   - Fall back to LLM extraction from the page content
2. Citation Formatting: Given metadata, format in any style:
   - APA 7th: Author, A. A. (Year). Title. Journal, Volume(Issue), Pages.
   - MLA 9th, Chicago, IEEE, Harvard
   - BibTeX: @article{key, author={...}, title={...}, ...}
3. Export Formats:
   - .bib file (BibTeX, importable by LaTeX, Zotero, Mendeley)
   - .ris file (importable by EndNote, RefWorks)
   - .json (CSL-JSON, used by Zotero and Citation.js)
   - Inline formatted text (copy-paste ready)
4. In-Report Citations: When generating research reports, automatically
   insert numbered inline citations [1][2] and append a formatted reference
   list at the bottom matching the user's preferred citation style.
5. Citation Deduplication: Detect when multiple sources point to the same
   paper (e.g., arXiv preprint + published version) and merge them.

Integration: New backend/core/skills/citation_manager.py. New API endpoint
GET /research/{session_id}/citations?format=bibtex. Frontend gets a
CitationExportButton on research results and a citation style selector in
user settings.

25. DIAGRAM AND ARCHITECTURE AUTO-GENERATOR
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot produces text-only outputs. When a user asks "Explain the
architecture of a transformer", a visual diagram would communicate the concept
far more effectively than paragraphs of text. Knowledge graph data also has
no visual export beyond the existing graph visualization.

What to Build: A DiagramGeneratorSkill that produces visual diagrams from
natural language or structured data:
1. Mermaid Diagrams: The LLM generates Mermaid.js syntax (flowcharts,
   sequence diagrams, class diagrams, entity-relationship diagrams, Gantt
   charts, state diagrams). The frontend renders them using the mermaid
   library (already a common npm package).
2. Architecture Diagrams: For system design questions, generate component
   diagrams showing services, databases, APIs, and data flows.
3. Concept Maps: From a set of findings or a knowledge graph subgraph,
   generate a concept map showing entities and relationships as a visual
   network. Uses D3.js force-directed layout or vis.js.
4. Comparison Tables: When comparing N items across M dimensions, auto-detect
   the comparison pattern and generate a structured comparison matrix with
   color-coded cells (green=advantage, red=disadvantage).
5. Timeline Diagrams: For historical or chronological queries, generate a
   visual timeline of events.
6. Auto-Detection: The system detects when a response would benefit from a
   diagram (presence of words like "architecture", "flow", "process",

   automatically.
7. Export: SVG and PNG download buttons on every generated diagram.

Integration: The LLM response post-processor in the orchestrator detects

wraps rendered diagrams with export controls.


─────────────────────────────────────────────────────────────────────────────────
26. PROMPT CHAINING AND WORKFLOW AUTOMATION
─────────────────────────────────────────────────────────────────────────────────

The Problem: Complex tasks require multiple sequential prompts — "Summarize
this paper, then extract the methodology, then compare it with paper X, then
write a critique." Users must manually chain these steps, copy-pasting output
from one query as input to the next.

What to Build: A WorkflowEngine that lets users define reusable multi-step
LLM pipelines:
1. Workflow Definition: A YAML/JSON schema defining steps:
   - name: "Paper Analysis Pipeline"
   - steps:
     - {id: "summarize", prompt: "Summarize: {{input}}", output_var: "summary"}
     - {id: "methods", prompt: "Extract methodology from: {{summary}}",
        output_var: "methods"}
     - {id: "critique", prompt: "Critique this methodology: {{methods}}",
        output_var: "critique"}
     - {id: "report", prompt: "Write a review combining: {{summary}},
        {{methods}}, {{critique}}", output_var: "final_report"}
2. Variable Interpolation: Each step can reference outputs from prior steps
   via {{variable_name}} syntax.
3. Conditional Branching: Steps can have conditions:
   if: "{{summary}}" contains "machine learning" → run ML-specific analysis;
   else → run general analysis.
4. Parallel Execution: Independent steps run concurrently via asyncio.gather.
5. Built-In Templates: Ship with 10+ pre-built workflows:
   - Paper Review Pipeline
   - Competitive Analysis Pipeline
   - Technical Due Diligence Pipeline
   - Content Repurposing Pipeline (research → blog → social posts → email)
   - Bug Report Triage Pipeline
6. Visual Builder: Frontend drag-and-drop workflow editor using
   react-flow or similar. Nodes represent steps, edges represent data flow.
7. Workflow Library: Save, share, and fork workflows. Export as JSON.

Integration: New backend/core/skills/workflow_engine.py with WorkflowRunner
class. New SQLite table: workflows. API endpoints: POST /workflow/create,
POST /workflow/run, GET /workflow/templates. Frontend gets WorkflowBuilder
and WorkflowRunnerPanel components.


─────────────────────────────────────────────────────────────────────────────────
27. PII DETECTION AND DATA ANONYMIZATION
─────────────────────────────────────────────────────────────────────────────────

The Problem: Users may ingest documents containing personally identifiable
information (PII) — names, emails, phone numbers, addresses, SSNs, credit
card numbers. This data gets embedded and stored in ArangoDB with no
redaction, creating privacy and compliance risks (GDPR, HIPAA, CCPA).

What to Build: A PIIGuard layer that runs during ingestion and optionally
during query responses:
1. Detection Engine: Use a combination of:
   - Regex patterns for structured PII (email, phone, SSN, credit card,
     IP addresses, dates of birth)
   - Named Entity Recognition (NER) via spaCy or a small transformer model
     for unstructured PII (person names, organization names, locations)
   - LLM-based detection for contextual PII ("the patient John Smith, age 42,
     from 123 Oak Street" — regex alone would miss the address context)
2. Redaction Modes:
   - Full redaction: Replace with [REDACTED] or type-specific placeholders
     ([EMAIL], [PHONE], [NAME])
   - Pseudonymization: Replace real names with consistent fake names
     (John Smith → Alex Johnson everywhere in the document) using Faker
     library. Maintain a reversible mapping stored encrypted.
   - Hash-based: Replace with a deterministic hash so the same entity always
     maps to the same token, preserving relationships without revealing
     identity.
3. Ingestion Gate: Before any document enters the embedding pipeline, PIIGuard
   scans it. If PII is detected, the user is notified with a preview of
   what would be redacted and can choose: redact, pseudonymize, or ingest
   as-is (with acknowledgment).
4. Response Guard: Optionally scan LLM responses before returning to the
   user, preventing the model from surfacing PII that exists in the
   knowledge base.
5. Audit Log: All PII detections and redaction decisions are logged for
   compliance auditing.

Integration: New backend/core/safety/pii_guard.py. Hooks into
MultimodalProcessor.process_file() as a pre-processing step. New SQLite
table: pii_audit_log. Frontend gets a PIIWarningModal that appears when
PII is detected during upload.


─────────────────────────────────────────────────────────────────────────────────
28. AGENTIC TOOL EXECUTOR — CALCULATOR, CODE RUNNER, API CALLER
─────────────────────────────────────────────────────────────────────────────────

The Problem: The LLM can reason about math, code, and data but cannot
actually execute anything. When asked "What's 17.3% of 4,829?", it does
mental math (and sometimes gets it wrong). When asked to run a Python
snippet, it can only generate code, not execute it.

What to Build: An AgentToolExecutor that gives the LLM access to real tools
via function calling:
1. Calculator Tool: Evaluates mathematical expressions with arbitrary
   precision (using Python's decimal module or sympy). Handles unit
   conversions (pint library), currency conversions (via free API), date
   arithmetic ("how many days between March 1 and December 25?").
2. Code Executor Tool: Runs Python code snippets in a sandboxed environment
   (RestrictedPython + resource limits: 5s timeout, 50MB memory, no network,
   no filesystem write). Returns stdout, stderr, and any generated plots.
3. Web Fetch Tool: Makes HTTP GET requests to user-approved URLs and returns
   the content. Useful for "What does this API endpoint return?" or "Fetch
   the latest exchange rate from this URL."
4. Knowledge Base Query Tool: The LLM can explicitly query its own knowledge
   base with a refined query, inspect the retrieved chunks, and decide
   whether to query again with different terms — true agentic retrieval.
5. File Writer Tool: Generates and saves files (CSV exports, JSON data,
   code files) to a user-accessible downloads directory.
6. Tool Selection: The orchestrator uses the LLM's function-calling
   capability (tool_use for Anthropic, functions for OpenAI) to let the
   model choose which tools to invoke, with what arguments, in what order.
   Supports multi-step tool chains (calculate → use result in code → plot).

Integration: New backend/core/skills/tool_executor.py with a ToolRegistry
class. Each tool is a callable with a JSON Schema description. The
orchestrator wraps tools into the LLM API call format. Frontend shows
ToolCallCard components displaying which tools were invoked and their
results.


─────────────────────────────────────────────────────────────────────────────────
29. SIDE-BY-SIDE COMPARATIVE ANALYSIS ENGINE
─────────────────────────────────────────────────────────────────────────────────

The Problem: Users frequently need to compare things — two papers, three
products, five frameworks — but Albot treats comparison queries the same as
any other query, producing a wall of prose. There is no structured comparison
output.

What to Build: A ComparisonEngineSkill that detects comparison intent and
produces structured, visual comparisons:
1. Intent Detection: Detect queries containing "compare", "vs", "versus",
   "difference between", "which is better", "pros and cons", or two or more
   named entities connected by "and" or "or".
2. Dimension Extraction: The LLM identifies the key comparison dimensions
   relevant to the entities being compared. E.g., comparing ML frameworks →
   dimensions: performance, ease of use, community size, documentation,
   GPU support, deployment options.
3. Per-Entity Analysis: For each entity, retrieve relevant information from
   the knowledge base and/or web search (if deep research is enabled).
   Extract scores or qualitative assessments for each dimension.
4. Comparison Matrix: Generate a structured comparison table with entities
   as columns and dimensions as rows. Each cell contains a concise
   assessment with optional color coding (green/yellow/red or numerical
   score).
5. Winner Summary: A brief narrative summarizing which entity is strongest
   in which areas, with a nuanced "best overall" recommendation that
   acknowledges trade-offs.
6. Radar Chart: Generate a radar/spider chart (matplotlib) showing each
   entity's profile across dimensions. Returned as inline image.
7. Persistent Comparisons: Save comparisons to a dedicated SQLite table so
   users can revisit and update them as new information becomes available.

Integration: New backend/core/skills/comparison_engine.py. The orchestrator
routes comparison-intent queries here. Frontend gets a ComparisonTableCard
and RadarChartCard component. API endpoint: POST /compare with body
{entities: [...], context: "..."}.

30. CONTENT MONITORING AND ALERT SYSTEM
─────────────────────────────────────────────────────────────────────────────────

The Problem: The knowledge base is static — it only grows when users manually
ingest documents. There is no way to say "watch arXiv for new papers about
diffusion models" or "alert me when a competitor publishes a new blog post."

What to Build: A ContentMonitor that continuously watches external sources:
1. Source Types:
   - RSS/Atom feeds (arXiv, blog feeds, news sites)
   - Web pages (detect changes via content hashing)
   - GitHub repositories (new releases, new issues)
   - Twitter/X accounts (via API or scraping)
   - Google Scholar alerts (author or keyword)
2. Match Rules: Users define rules combining:
   - Keyword match (any of: "diffusion", "score-based")
   - Semantic similarity (embedding cosine > 0.7 with a reference query)
   - Source filter (only from arxiv.org, only from specific authors)
   - Recency (only content published in the last 7 days)
3. Actions on Match:
   - Auto-ingest: Automatically add the matched content to the knowledge base.
   - Alert: Send an in-app notification, email, or webhook.
   - Digest: Batch matches and send a daily/weekly summary.
   - Deep Research: Automatically trigger a deep research session on the
     matched topic (for high-priority watches).
4. Deduplication: Content fingerprinting (SimHash) prevents the same content
   from being ingested or alerted multiple times.
5. Dashboard: A monitoring dashboard showing all active watches, recent
   matches, and match history.

Integration: New backend/core/monitoring/content_monitor.py using APScheduler
for periodic checks and feedparser for RSS. New SQLite tables:
content_watches, content_matches. Frontend gets MonitoringDashboard component.

31. BROWSER EXTENSION — CLIP, ANNOTATE, AND INGEST FROM THE WEB
─────────────────────────────────────────────────────────────────────────────────

The Problem: Ingesting web content requires copying URLs, switching to Albot,
and pasting them. There is no frictionless way to capture content while
browsing.

What to Build: A lightweight Chrome/Firefox extension:
1. Clip to Albot: Right-click on any webpage → "Save to Albot". The extension
   extracts the page content (using Mozilla Readability.js for article
   extraction), sends it to POST /ingest/web-clip with the URL, title, and

2. Highlight and Save: Select text on any page → right-click → "Save
   highlight to Albot". Saves the selected text with the source URL as a
   fine-resolution KnowledgeAtom, tagged with the highlight context.
3. Quick Ask: Select text → right-click → "Ask Albot about this". Opens a
   small popup with a chat interface pre-populated with the selected text
   as context. Uses the existing /chat endpoint.
4. PDF Capture: When viewing a PDF in the browser, a floating "Ingest in
   Albot" button appears. Click to send the PDF URL to
   POST /ingest/url for processing.
5. Sidebar Panel: A persistent sidebar that shows the Albot chat interface
   alongside any webpage. Users can ask questions about the page they are
   currently viewing.
6. Auto-Capture Mode: Optionally, the extension can auto-capture every page
   the user visits (with domain whitelist/blacklist) for passive knowledge
   base building.

Integration: Extension source in a new /extension directory. Backend gets a
new POST /ingest/web-clip endpoint that accepts {url, title, content,
highlights: [...]}. The extension communicates with the backend via the
existing API, authenticated with an API key stored in extension settings.

32. PRESENTATION BUILDER — RESEARCH TO SLIDES
─────────────────────────────────────────────────────────────────────────────────

The Problem: After a deep research session, users often need to present their
findings. They manually copy-paste from the research report into PowerPoint.
There is no automated pipeline from research output to presentation.

What to Build: A PresentationBuilderSkill:
1. Report-to-Slides: Given a research report (from deep research or a chat

34. LLM COST DASHBOARD AND BUDGET MANAGEMENT
─────────────────────────────────────────────────────────────────────────────────

The Problem: The LLM router transparently calls multiple providers (Groq,
OpenAI, Anthropic, Gemini, OpenRouter) with fallback chains, but there is
zero visibility into costs. A heavy deep research session might cost $2 or
$0.02 depending on which models were used and how many tokens were consumed.
Users have no way to set budgets or track spending.

What to Build: A CostTracker integrated into LLMRouter:
1. Token Counting: After every LLM call (complete and async_complete), log:
   provider, model, input_tokens, output_tokens, timestamp, caller
   (chat/research/ingestion/judge), session_id.
2. Cost Calculation: Maintain a price table for each provider/model
   (e.g., GPT-4o: $2.50/$10.00 per 1M input/output tokens; Claude Sonnet:
   $3/$15; Groq Llama: free). Calculate cost per call.
3. Session Cost Rollup: Aggregate costs per chat session, per deep research
   session, per day, per week, per month. Store in SQLite.
4. Budget Limits: Users can set:
   - Per-session budget: "Stop research if cost exceeds $1.00"
   - Daily budget: "Cap all LLM usage at $5.00/day"
   - Monthly budget: "Alert me at $50, hard stop at $100"
   When a limit is hit, the system gracefully stops (early synthesis for
   research, a warning message for chat) rather than silently failing.
5. Cost Optimization Suggestions: The dashboard shows which operations are
   most expensive and suggests optimizations: "Deep research accounts for
   83% of your spend. Consider reducing max_sources from 40 to 25."
6. Provider Comparison: Show cost-per-quality metrics by comparing LLM
   judge scores against cost per session, helping users choose optimal
   provider configurations.

Integration: CostTracker class added to backend/core/llm/cost_tracker.py.
LLMRouter.complete() and async_complete() call tracker.log() after each
invocation. New API endpoint: GET /analytics/costs?period=weekly. Frontend
gets a CostDashboardPage with charts (daily spend line chart, provider
breakdown pie chart, top-sessions bar chart).


─────────────────────────────────────────────────────────────────────────────────
35. PLUGIN AND EXTENSION FRAMEWORK
─────────────────────────────────────────────────────────────────────────────────

The Problem: Every new capability requires modifying core backend code. There
is no way for users or third-party developers to add custom skills, data
sources, or output formats without forking the entire codebase.

What to Build: A PluginFramework that allows modular capability extension:
1. Plugin Interface: A base class AlbotPlugin with lifecycle hooks:
   - on_install(): Run setup (create tables, download models)
   - on_query(query, context) -> Optional[Response]: Intercept queries
     if the plugin can handle them (return None to pass through)
   - on_ingest(document) -> Optional[List[KnowledgeAtom]]: Custom
     ingestion for new file types
   - on_post_process(response) -> Response: Modify responses before
     delivery (add citations, translate, redact)
   - get_tools() -> List[ToolDefinition]: Register tools for the
     agentic executor
2. Plugin Manifest: Each plugin is a directory with a plugin.yaml:
   name, version, description, author, dependencies, permissions
   (which APIs it can access, whether it needs network, filesystem).
3. Plugin Manager: Install plugins from a local directory, a git repo URL,
   or a future plugin registry. Handles dependency resolution, version
   conflicts, and sandboxing.
4. Permission System: Plugins declare required permissions. Users approve
   permissions on install. Plugins cannot access APIs they were not granted.
5. Plugin SDK: A lightweight Python package (albot-plugin-sdk) that provides
   the base classes, type definitions, and testing utilities for plugin
   developers.
6. Hot Reload: Plugins can be installed, updated, and uninstalled without
   restarting the server.

Integration: New backend/core/plugins/ module with plugin_manager.py,
plugin_base.py, plugin_registry.py. The orchestrator's query routing checks
registered plugins before falling back to built-in skills. Frontend gets a
PluginManagerPage showing installed plugins with enable/disable toggles.


─────────────────────────────────────────────────────────────────────────────────
36. PERSONA AND ROLE-BASED AGENT PROFILES
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot has a single personality and communication style. A PhD
student, a startup CEO, and a high school teacher all get the same tone,
vocabulary level, and response structure.

What to Build: A PersonaSystem that lets users select or create AI agent
profiles:
1. Built-In Personas:
   - Research Assistant: Formal, thorough, citation-heavy, academic tone
   - Technical Advisor: Concise, code-heavy, assumes expertise
   - Explainer: Simple language, analogies, step-by-step, ELI5-friendly
   - Executive Briefer: Business language, bottom-line-up-front, action items
   - Socratic Tutor: Answers questions with questions, guides discovery
   - Creative Writer: Narrative style, vivid language, storytelling
   - Devil's Advocate: Challenges assumptions, presents counterarguments
2. Custom Personas: Users create their own with:
   - Name and avatar
   - System prompt template
   - Preferred response length (concise / standard / detailed)
   - Vocabulary level (simplified / standard / technical / academic)
   - Interaction style (directive / collaborative / socratic)
   - Knowledge base scope (all / specific namespaces / specific documents)
3. Persona Memory: Each persona maintains its own conversation style
   preferences, learned from user feedback ("too verbose" → adjusts down).
4. Multi-Persona Chat: In a single session, switch between personas for
   different perspectives. E.g., ask the Research Assistant to analyze, then
   switch to Devil's Advocate for critique.
5. Persona Templates: Export/import persona configs as JSON for sharing.

Integration: New backend/core/skills/persona_system.py. Persona configs
stored in SQLite personas table. The orchestrator prepends the active
persona's system prompt to every LLM call. Frontend gets a PersonaSelector
component in the chat header and a PersonaEditorModal.

37. ANNOTATION AND HIGHLIGHTING LAYER FOR INGESTED DOCUMENTS
─────────────────────────────────────────────────────────────────────────────────

The Problem: Once a document is ingested, users cannot interact with it
visually. There is no way to highlight a paragraph in an ingested PDF and say
"explain this" or "this is important, remember it." The document becomes an
opaque bag of embeddings.

What to Build: An AnnotationSystem:
1. Document Viewer: A built-in document viewer (PDF.js for PDFs, custom
   renderer for DOCX/text) that displays the original document in the
   frontend.
2. Inline Highlighting: Users can select text spans and apply color-coded
   highlights (yellow=important, red=question, green=agree, blue=reference).
3. Annotations: Each highlight can have an attached note (free text or
   structured tags).
4. Ask About Highlight: A "Ask Albot" button on each highlight sends the
   highlighted text as context to the chat, pre-populated with: "Explain
   this passage: [highlighted text]" or "What does this mean in the context
   of [document title]?"
5. Importance Boosting: Highlighted passages get their embedding importance
   score boosted in the vector store, making them more likely to be
   retrieved in future queries. Yellow highlights boost by 1.5x, multiple
   highlights on the same passage stack.
6. Annotation Search: A search interface across all annotations —
   "Show me everything I highlighted about attention mechanisms."
7. Annotation Export: Export all annotations for a document as a structured
   document (markdown, JSON, or a separate annotated PDF with highlights
   rendered).

Integration: Frontend gets a DocumentViewer component using react-pdf (for
PDF) and a custom AnnotationOverlay. Backend gets new SQLite tables:
annotations, highlights. API endpoints: POST /annotations/create,
GET /annotations/search, GET /documents/{id}/view. The retrieval engine
reads highlight boost scores during query time.

38. AUTOMATED MEETING NOTES — TRANSCRIBE, SUMMARIZE, ACTION ITEMS
─────────────────────────────────────────────────────────────────────────────────

The Problem: Users already upload audio files for transcription via Whisper,
but the output is a raw transcript. There is no structured meeting intelligence
— no summary, no action items, no speaker identification, no topic
segmentation.

What to Build: A MeetingAssistantSkill that transforms audio/video recordings

1. Enhanced Transcription: Use Whisper with word-level timestamps. For longer

2. Speaker Diarization: Use pyannote.audio (open-source, HuggingFace) to

   - Key decisions made

   namespace="meetings", so future queries can reference past meeting

7. Follow-Up Generation: After processing, the LLM generates a follow-up
   email draft summarizing the meeting and listing action items, ready to
   send.

Integration: New backend/core/skills/meeting_assistant.py. Uses existing
Whisper model. Adds pyannote.audio for diarization. New SQLite tables:
meetings, action_items. Frontend gets MeetingPanel with topic timeline,

40. KNOWLEDGE BASE HEALTH DASHBOARD AND AUTO-MAINTENANCE
─────────────────────────────────────────────────────────────────────────────────

The Problem: As the knowledge base grows (hundreds of documents, thousands of
chunks), quality degrades silently. Stale content, duplicate embeddings,
orphaned nodes, and embedding drift accumulate without any visibility.

What to Build: A KBHealthDashboard:
1. Health Metrics (computed daily or on-demand):
   - Total documents, chunks, embeddings, graph nodes, graph edges
   - Duplicate detection: Clusters of near-duplicate chunks (cosine > 0.95)
   - Orphan detection: Graph nodes with zero edges (isolated knowledge)
   - Staleness score: Documents older than N months with no queries hitting
     them (zero retrieval hits in the access log)
   - Embedding coverage: Percentage of chunks that have valid embeddings
     (detect corrupted or missing vectors)
   - Namespace balance: Distribution of content across namespaces
2. Auto-Maintenance Tasks:
   - Deduplicate: Merge near-duplicate chunks, keeping the one with higher
     importance and more graph connections
   - Re-embed: Re-generate embeddings for chunks whose embeddings are from
     an older model version (model upgrade detection)
   - Prune: Flag stale, never-retrieved content for user review and optional
     deletion
   - Re-index: Rebuild ArangoDB indexes when fragmentation exceeds threshold
   - Graph Repair: Connect orphaned nodes to their nearest semantic neighbors
3. Quality Score: A single 0-100 "Knowledge Base Health Score" combining all
   metrics, displayed prominently on the dashboard.
4. Scheduled Maintenance: Auto-run maintenance tasks weekly via APScheduler,
   with a summary notification of what was cleaned.

Integration: New backend/core/maintenance/kb_health.py. API endpoint:
GET /analytics/kb-health. Frontend gets a KBHealthDashboard page with
metric cards, trend charts, and one-click maintenance action buttons.

42. EXPORT AND BACKUP SYSTEM — FULL KNOWLEDGE PORTABILITY
─────────────────────────────────────────────────────────────────────────────────

The Problem: All knowledge, chat history, research sessions, and memory are

What to Build: A comprehensive ExportAndBackupSystem:

   - Configuration files (.env settings, research configs)

45. INTELLIGENT CHUNKING STRATEGIES — ADAPTIVE DOCUMENT DECOMPOSITION
─────────────────────────────────────────────────────────────────────────────────

The Problem: The multimodal processor uses naive sentence-splitting for text
and fixed-size chunking for other formats. A 50-page legal contract and a
2-page blog post both get the same treatment. This produces either too-small
chunks that lose context or too-large chunks that dilute relevance.

What to Build: An AdaptiveChunker that selects the optimal chunking strategy
per document:
1. Document Classification: Before chunking, classify the document type
   (academic paper, legal contract, technical docs, narrative text, code,
   Q&A/FAQ, tabular data) using a fast LLM call or rule-based heuristics.
2. Strategy Selection:
   - Academic papers: Section-aware chunking using heading detection.
     Abstract, Introduction, Methods, Results, Discussion become separate
     chunks. References are extracted and linked, not chunked.
   - Legal contracts: Clause-level chunking. Each numbered clause/sub-clause
     becomes a chunk. Cross-references between clauses create graph edges.
   - Technical documentation: Topic-based chunking using heading hierarchy.
     Code blocks are kept intact within their parent chunk.
   - Narrative text: Sliding window with semantic boundary detection —
     split at paragraph breaks where topic changes (detected by embedding
     distance between consecutive paragraphs exceeding threshold).
   - Q&A/FAQ: Each question-answer pair becomes a single chunk.
   - Tabular data: Row-group chunking (keep related rows together based
     on a key column).
3. Chunk Size Optimization: Target chunk sizes based on the embedding
   model's optimal input length. For E5-Large: 256-512 tokens. Chunks
   below 50 tokens are merged with neighbors. Chunks above 1024 tokens
   are split at the nearest sentence boundary.
4. Overlap Windows: Configurable overlap (default 10%) between adjacent
   chunks to prevent information loss at boundaries.
5. Chunk Quality Score: After embedding, compute a "chunk coherence score"
   (intra-chunk cosine similarity). Flag low-coherence chunks for
   re-splitting.

Integration: Replace the current process_text_file() logic in
MultimodalProcessor with a call to AdaptiveChunker.chunk(content, doc_type).
New backend/core/ingestion/adaptive_chunker.py. The document classifier is
a lightweight wrapper around the LLM with a cached classification prompt.

46. RESPONSE VERSIONING AND A/B COMPARISON
─────────────────────────────────────────────────────────────────────────────────

The Problem: When a user asks the same question at different times, they get
different answers (due to knowledge base changes, model updates, or
temperature). There is no way to compare response versions or understand
why the answer changed.

What to Build: A ResponseVersioning system:
1. Response Fingerprinting: Each response is hashed (content + sources used +
   model used) and stored with its full generation context.
2. Query Re-Run: A "Re-ask" button on any past response re-runs the same
   query with the current knowledge base state. The new response is shown
   side-by-side with the original, with diff highlighting.
3. A/B Model Comparison: A "Compare Models" mode where the same query is
   sent to two different LLM providers simultaneously. Both responses appear
   side-by-side with the LLM Judge scoring both. Helps users find their
   optimal model configuration.
4. Response History: For each unique query (normalized), keep a version
   history of all responses, timestamps, and which knowledge base state
   produced each one. A timeline view shows how the answer evolved.
5. Regression Detection: Automated detection when a re-asked query gets a
   significantly worse LLM Judge score than previously — indicates knowledge
   base degradation or model regression.
6. Source Attribution Diff: Show which sources were used in each version.
   Highlight newly added sources and removed sources.

Integration: New backend/core/analytics/response_versioning.py. New SQLite
table: response_versions. Frontend gets ResponseDiffViewer and
ModelComparisonPanel components. API endpoints: POST /compare/models,
GET /responses/{query_hash}/versions.

48. RETRIEVAL CONFIDENCE SCORING AND ANSWER CALIBRATION
─────────────────────────────────────────────────────────────────────────────────

The Problem: Albot answers every question with equal confidence, whether it
has 10 highly relevant sources or zero. Users cannot tell if the answer is
well-supported or hallucinated. The system gives no signal of its own
uncertainty.

What to Build: A ConfidenceCalibration system:
1. Retrieval Confidence: After retrieval, compute a confidence score based on:
   - Top-K retrieval scores (are the top results highly similar or barely
     matching?)
   - Score gap between top-1 and top-5 (large gap = clear best match;
     small gap = ambiguous)
   - Number of sources that independently support the same claim
   - Knowledge base coverage (is the query topic well-represented in the KB?)
2. Answer Confidence: After LLM generation, run a self-consistency check:
   - Generate 3 responses at temperature=0.7
   - Compute semantic similarity between all 3
   - High similarity = high confidence (the model converges on the same
     answer regardless of sampling)
   - Low similarity = low confidence (the answer is unstable)
3. Confidence Display: Show a confidence indicator next to each response:
   - High (green): Well-supported by multiple sources, consistent answers
   - Medium (yellow): Some support, but gaps or ambiguity detected
   - Low (red): Weak retrieval, inconsistent answers, or out-of-domain query
4. Calibrated Caveats: When confidence is medium or low, the system
   automatically prepends a caveat: "Note: This answer is based on limited
   sources and may not be fully accurate." or "I'm not confident about this
   — consider verifying with additional sources."
5. Confidence Analytics: Track confidence distributions over time. Identify
   topic areas where confidence is consistently low (knowledge base gaps).
6. Abstention: When confidence is critically low (below threshold), instead
   of hallucinating, the system says: "I don't have enough information to
   answer this reliably. Consider running a deep research session or
   ingesting relevant documents."

Integration: New backend/core/retrieval/confidence_scorer.py. Called after
retrieval and after LLM generation. The ConfidenceScore is attached to the
response object and rendered in the frontend as a ConfidenceBadge component.
